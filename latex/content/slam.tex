\chapter{Simultaneous Localization and Mapping~(SLAM) in Dynamic Environments}\label{chap:slam}

This chapter presents RicoSLAM, a graph-based 2D~\gls{slam} framework for dynamic
environments developed within the scope of this PhD~Thesis, building on the 2D
point cloud registration framework introduced in \chapref{chap:distance-maps}.
RicoSLAM integrates the distance map-based tracker from
\chapref{chap:distance-maps} as its front-end with the
srrg2\_solver~\cite{grisetti:robotics:2020} optimization back-end to construct
an $SE(2)$ pose graph, detecting loop closures by combining nodes' proximity
in the world frame with graph-distance of the loop chain.
The proposed \gls{slam} framework filters dynamic observations
online during local map maintenance using a 2D laser scanner ray model,
while providing an offline 2D occupancy grid map generation step in which
the same ray model is combined with
Bresenham-like~\cite{bresenham:ibm:1965,zingl:uas:2016} line rasterization to
ignore dynamic observations when projecting all scans considered during online
\gls{slam} onto the final grid map. Additionally, this chapter presents
novel formulations for \gls{gba} based on distance map-based per-point factors,
building on the point-to-point and point-to-plane distance map-based
formulations presented in \chapref{chap:distance-maps}.
Experimental results indicate that RicoSLAM can operate in highly dynamic,
noisy environments while filtering dynamic observations from
the 2D~occupancy grid representation of the environment.

This chapter is organized as follows.
First, \secref{sec:slam:ricoslam} presents a high-level overview of the RicoSLAM
framework, positioning the contributions of this chapter in relation to
the components inherited from \chapref{chap:distance-maps}.
Next, \secref{sec:slam:online} describes the online \gls{slam} pipeline,
covering the $SE(2)$ pose-pose geodesic error factor and its Jacobians via
right-sided perturbation available in the
srrg2\_solver~\cite{grisetti:robotics:2020} framework, the local map splitting
and merging criteria, the loop closure detection procedure, the relocalization
mechanism, and the pose history streams (laser odometry, online and offline
corrected poses) logged for trajectory evaluation.
\secref{sec:slam:dynamics} describes the dynamics removal strategy proposed in
this work, both online (during local map maintenance) and offline
(during 2D~occupancy grid map generation).
Moreover, \secref{sec:slam:gba} introduces the distance map-based \gls{gba}
per-point formulations proposed in this work, deriving the
point-to-point and point-to-plane error factors together with their Jacobians
via right-sided perturbation on the $SE(2)$ manifold.
Then, \secref{sec:slam:results} presents the experimental evaluation, including
a benchmark on the IILABS~3D~\cite{ribeiro:access:2025} dataset in terms of
accuracy and scalability, the impact of dynamics removal from the 2D occupancy
grid representation, and a synthetic benchmark to evaluate the distance
map-based \gls{gba} proposed in this work.
Lastly, \secref{sec:slam:conclusions} summarizes the main conclusions and
directions for future work.

\section{Framework Overview}\label{sec:slam:ricoslam}

\begin{figure}[t]
  \centering
  \includegraphics[width=\textwidth]{slam/system/system-overview.drawio.pdf}
  \caption[Block diagram of the RicoSLAM framework]{%
  Block diagram of the RicoSLAM framework.}
  \label{fig:slam:ricoslam}
\end{figure}

RicoSLAM is a 2D~graph \gls{slam} framework compatible with 2D~laser scanners
and wheeled odometry for dynamic environments in industrial \gls{amr} applications.
This framework represents the robot trajectory and the environment map as
a sparse factor graph~\cite{grisetti:itsm:2010,dellaert-kaess:ftr:2017}.
Each $SE(2)$ variable in the graph encodes the pose of a keyframe in
a common world frame, together with its measurement:
the locally built point cloud, the precomputed distance map representation
introduced in \secref{sec:distance-maps:alignment:dmaps}, and
the 2D~laser scanner messages aggregated within that keyframe. Each factor
encodes a relative-pose measurement between two keyframes, generated either by
odometric chaining of the front-end tracker or by loop closure detection.

\figref{fig:slam:ricoslam} presents a block diagram of RicoSLAM.
Internally, the system is organized around two top-level containers -- a tracker
front-end and a factor graph back-end -- with two additional offline
post-processing stages, a distance map-based \gls{gba} procedure and a
2D~occupancy grid mapper, that operate on the optimized pose graph once the
online run has completed. All the blocks in the diagram are summarized as follows:
\begin{itemize}
\item The tracker is built on top of the distance map-based aligner from
      \chapref{chap:distance-maps} and is extended in this chapter with a
      dynamics-aware local mapping stage. Preprocessing
      (conversion of laser scan messages into a point cloud representation,
      voxelization, and normal computation) and the distance map-based
      point-to-point and point-to-plane formulations
      (see~\secref{sec:distance-maps:alignment:pt2pt} and
      \secref{sec:distance-maps:alignment:pt2pl}) are inherited from
      \chapref{chap:distance-maps}. We also added to the framework both
      point-to-point~\cite{besl-mckay:tpami:1992} and
      point-to-plane~\cite{chen-medioni:icra:1991} \gls{icp} formulations for
      2D~point cloud registration with the tracker's aligner,
      using distance map-guided data association.
      New to this chapter are the split and merge triggering criteria,
      which monitor the aligner output stream
      (translation, orientation, viewpoint, inlier ratio, and the
      timestamp from the latest laser scan message processed by the tracker).
      The split criterion determines when to commit the current keyframe to
      the graph and start tracking on a new keyframe.
      The merge criterion determines when to merge new returns into
      the tracker's local map, using a merger container that performs
      online ray-based dynamics removal on that local map by treating
      each laser beam as a sensor model that may see through a previously
      mapped point or add new observations to the local representation.
      The tracker maintains the current keyframe --
      local point cloud, precomputed distance map, and array of raw laser scans
      aggregated since the last split -- and exposes its
      laser odometry estimate and alignment statistics to the back-end;

\item The factor graph back-end uses srrg2\_solver~\cite{grisetti:robotics:2020}
      to solve the optimization problem and host all the components that operate
      on the graph. Its core data structure is the pose graph,
      populated by the tracker through the update-on-split action that
      promotes the current keyframe to a graph variable and adds a geodesic
      factor against the previously committed keyframe.
      The loop finder and graph visitor containers traverse the graph
      in search of loop closure candidates by combining world-frame proximity
      around the current robot pose with a graph-distance constraint that
      ensures non-trivial loop geometry. Each candidate is validated by
      the loop aligner, aligning the latest point cloud processed by the tracker
      against the candidate keyframe's local map, using two complementary
      initial guesses (pose graph-based and same-place-based). This loop aligner
      may be the same as the tracker's front-end or a different one.
      After a successful loop closure, a relative-pose geosedic factor is added
      between the current keyframe and the loop variables. Also, if
      the relocalization policy is enabled, it updates the tracker's keyframe
      from an existing one on the graph, relocalizing the robot in
      the existing back-end graph. The Gauss-Newton solver, provided by
      srrg2\_solver~\cite{grisetti:robotics:2020} and the only third-party
      component of the framework, performs iterative least-squares optimization
      on the $SE(2)$ manifold with a sparse-block Cholesky linear solver,
      and is invoked incrementally whenever a loop closure factor is added to
      the graph. Furthermore, a pose history container makes available the three
      pose streams produced over time: the raw laser odometry from the tracker,
      the online \gls{slam} pose estimate, and the corrected pose based on
      the online estimate while considering the loop closures throughout time.
      RicoSLAM generates TUM~\cite{sturm:iros:2012} files for these three pose
      streams through external \gls{ros} services upon user request.
      Additionally, RicoSLAM saves the factor graph through the serialization
      framework made available through the
      srrg2\_solver~\cite{grisetti:robotics:2020} into a human-readable JSON
      file, including keyframe variables and their global poses,
      the $SE(2)$ geosedic pose-pose factors, 2D~point clouds, and
      the arrays of 2D~laser scan messages associated with each keyframe;

\item Finally, two offline post-processing stages consume the optimized pose
      graph from the serialized JSON file after the online run completes.
      The \gls{gba} replaces the pose-pose constraints obtained from
      the tracker and the loop aligner by per-scan-point distance map residuals
      between different laser scan messages, thereby refining all scans that
      contributed to the online \gls{slam} map through novel distance map-based
      \gls{gba} formulations proposed in this chapter.
      The 2D~occupancy grid mapper generates a ROS-compatible occupancy grid map
      by Bresenham-based~\cite{bresenham:ibm:1965,zingl:uas:2016} rasterization
      of each retained scan, using the same laser ray model as
      the online merger to ignore the contributions of dynamic observations.
\end{itemize}

Throughout this chapter, the notation introduced in
\secref{sec:distance-maps:alignment:problem} is kept whenever possible.
Consistent with the convention adopted in \chapref{chap:distance-maps}
for $SE(2)$ optimization variables, a keyframe variable in the pose graph
is denoted by $\mathbf{x} \in SE(2)$ when it refers to a generic graph node,
and is given subscripts \emph{from} or \emph{to} when both endpoints of
a pose-pose geosedic factor must be distinguished.
The current pose \gls{wrt} the world frame is denoted
$\mathbf{x}_{R,W} = \mathbf{x}_{\mathrm{kf}} \cdot \mathbf{x}_{R,\mathrm{kf}}$,
where $\mathbf{x}_{\mathrm{kf}}$ is the current keyframe estimate and
$\mathbf{x}_{R,\mathrm{kf}}$ is the local laser odometry estimate of
the robot \gls{wrt} that keyframe. The right-sided perturbation update
$\mathbf{x} \oplus \boldsymbol{\xi} = \mathbf{x}\cdot\exp(\boldsymbol{\xi})$
adopted in
\secref{sec:distance-maps:background:least-squares:manifold} and
\secref{sec:distance-maps:alignment:problem} is used in all error and
Jacobian derivations of this chapter.

\subsection{Processing pipeline}\label{sec:slam:online:overview}

The tracker front-end of RicoSLAM processes the 2D~laser scan message stream
from the sensor and, optionally, wheel odometry data for initialization during
the point cloud registration step. \algref{alg:slam:online:tracker} summarizes
the processing pipeline for the latest scan message ($\mathcal{S}$), updating
the local robot pose \gls{wrt} the current keyframe
($\mathbf{x}_{R,\mathrm{kf}}$) and providing a laser odometry estimate to
the RicoSLAM back-end. First, the tracker converts the range vector of the
incoming laser scan ($\mathcal{S}$) into a 2D~point cloud ($\mathcal{P}$) in the
sensor frame using the scanner's angular parametrization, preserving the data ordering.
This conversion marks points whose ranges fall outside the configured admissible
range interval or outside the sensor's minimum and maximum ranges as invalid,
so the aligner ignores them. Two additional preprocessing steps include
computing the point cloud normals
(see~\secref{sec:distance-maps:alignment:pt2pl:normals}) and voxelizing
the point cloud using a given voxelization scale ($r_v$), or assuming
the same resolution as the distance map representation.
Next, the tracker updates $\mathbf{x}_{R,\mathrm{kf}}$ using wheel odometry data
if the user enables it. If not, the tracker assumes the robot's pose from
the previous scan message for aligner initialization. In the case that it is
the first laser scan message processed by the tracker, the latter initializes
the local map ($\mathcal{M}_{\mathrm{kf}}$), building its distance map
representation and appending the first laser scan message, and initializes the
local robot pose as an identity $SE(2)$ pose, changing the state of
the tracker to \emph{Initialized}.

\input{algorithms/slam/tracker.tex}

Then, the tracker's aligner estimates the local robot pose $\mathbf{x}_{R,\mathrm{kf}}$
that best aligns the moving cloud ($\mathcal{P}$) from the latest scan message ($\mathcal{S}$)
to the reference cloud of the current keyframe. This aligner implements the
distance map-based formulations proposed in \chapref{chap:distance-maps}, with
the option also to use \gls{icp} point-to-point~\cite{besl-mckay:tpami:1992} and
point-to-plane~\cite{chen-medioni:icra:1991} formulations.
If the aligner fails the registration process, the tracker resets its
local map ($\mathcal{M}_{\mathrm{kf}}$) and local robot pose
($\mathbf{x}_{R,\mathrm{kf}}$), setting the internal state to \emph{Error}.
After updating $\mathbf{x}_{R,\mathrm{kf}}$ and its information matrix
($\boldsymbol{\Omega}_{R,\mathrm{kf}}$) upon a successful registration,
the tracker changes its internal state to \emph{Tracking}.
Afterward, the tracker first evaluates whether the split criterion is triggered
based on the aligner stats ($\chi_{\mathrm{inlier}}^2$, inlier ratio),
$\mathbf{x}_{R,\mathrm{kf}}$, and the timestamp of the latest scan message.
If the split criterion is triggered, the tracker changes its current keyframe
and sets an $SE(2)$ identity pose, changing the internal state to
\emph{TrackingMeasurementSplit}. Second, if the split criterion is not triggered
and the merge criterion is triggered based on the same evaluation as for
the former, the tracker's merger fuses the moving cloud ($\mathcal{P}$) into
the local map ($\mathcal{M}_{\mathrm{kf}}$) using the online ray-based dynamics
removal model (see~\secref{sec:slam:dynamics:online}).
The raw scan is appended to the keyframe's history of aggregated measurements,
changing the internal state to \emph{TrackingMeasurementMerged}.
Testing splitting before merging ensures that a scan satisfying both criteria is
committed as a new keyframe rather than merged into an outgoing one.
The five states admissible by the tracker --
\emph{Initialized}, \emph{Error}, \emph{Tracking},
\emph{TrackingMeasurementSplit}, and \emph{TrackingMeasurementMerged} --
are then used by the back-end of RicoSLAM to decide whether to
insert a fixed keyframe, an odometric pose-pose factor, or simply
update the corrected-pose stream.

Next, \algref{alg:slam:online} summarizes the online processing pipeline
triggered when a new scan message ($\mathcal{S}$) arrives at
the RicoSLAM system. First, as already explained, the tracker processes
the message stream and the wheel odometry data
(see~\algref{alg:slam:online:tracker}). Depending on the tracker's status --
\emph{Initialized} (first scan after the tracker initialized its current
keyframe), \emph{Error} (registration failure), or
\emph{TrackingMeasurementSplit} (the splitting criterion fired making
the tracker changing its current keyframe) -- a new graph node may be inserted
with either a fixed status (reserved for the first keyframe variable of
the back-end factor graph) or an active status with an odometric pose-pose
factor linking it to the previous keyframe. The distinction between
fixed and active variables is what makes the pose graph optimization problem
well-posed. The cost minimized by the Gauss-Newton solver is a sum of
geodesic errors of the form
$\left(\mathbf{x}_i^{-1}\cdot\mathbf{x}_j\right)\ominus\mathbf{z}_{ij}$,
which depend only on the relative pose between the two endpoints
(see~\secref{sec:literature:slam:graph}). Applying any transformation
$\mathbf{g}\in SE(2)$ to every variable in the graph leaves unchanged
these relative quantities their the cost. Consequently,
the Gauss-Newton approximate Hessian
$\mathbf{H} = \mathbf{J}^{\top}\boldsymbol{\Omega}\mathbf{J}$ is
rank-deficient and any trajectory would be determined only up to
a global rigid-body transformation. Marking the first keyframe as
fixed eliminates its three columns from the normal equations, makes the reduced
Hessian full-rank, and defines the world frame the same as that
keyframe~\cite{grisetti:itsm:2010,grisetti:robotics:2020}.

\input{algorithms/slam/slam.tex}

After the keyframe-management step, the back-end of RicoSLAM updates
the robot pose \gls{wrt} the world frame ($\mathbf{x}_{R,W}$)
by composing the current keyframe estimate in the factor graph
($\mathbf{x}_{\mathrm{kf}}$) with the local laser odometry of the robot
\gls{wrt} that keyframe ($\mathbf{x}_{R,\mathrm{kf}}$), as formulated in
\eqref{eq:slam:online:robot-in-world}.
\begin{equation}
\label{eq:slam:online:robot-in-world}
\mathbf{x}_{R,W} = \mathbf{x}_{\mathrm{kf}}\cdot\mathbf{x}_{R,\mathrm{kf}}
\end{equation}

If loop closure detection is enabled, the back-end of RicoSLAM invokes the loop
finder to compute the set of candidate hints
(see~\secref{sec:slam:online:loop-closure}).
Each hint is then validated by running the loop aligner (either the same
distance map-based aligner used by the tracker or a dedicated instance)
on the current moving point cloud against the candidate keyframe's local
map. Successfully aligned hints generate loop closure factors that are added
to the graph, based on the loop aligner estimation of the robot \gls{wrt} the
loop keyframe ($\hat{\mathbf{x}}_{R,\ell}$) and the robot \gls{wrt} the current
one ($\mathbf{x}_{R,\mathrm{kf}}$).
If at least one loop factor is added,
the solver runs a single optimization pass and the world-frame robot pose
is updated accordingly (see \eqref{eq:slam:online:robot-in-world}).
Finally, if relocalization is enabled and at least one
short-loop candidate is available, the relocalization policy described in
\secref{sec:slam:online:relocalization} may transfer the tracker into an
existing graph node instead of growing the graph further.
